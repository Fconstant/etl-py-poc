What Is BigQuery?

You've spent years building apps with Postgres. You know rows, indexes, connection pools, VACUUM. BigQuery is a different beast. This lesson builds the mental model.

The One-Liner

BigQuery is Google Cloud's serverless data warehouse. You throw data at it, write SQL, and it gives you answers — fast, at any scale, with zero infrastructure.

BigQuery vs What You Know

As a fullstack dev, your mental model is probably:

App → Postgres (row store, OLTP) → App
     ↑ writes rows one at a time
     ↑ reads specific records by ID/index

BigQuery flips this:

Data sources → BigQuery (column store, OLAP) → Analysts/Dashboards
              ↑ bulk loads (thousands to billions of rows)
              ↑ scans entire columns across millions of rows
Key Insight

Postgres is optimized for OLTP (online transactional processing) — fast writes, fast lookups by primary key, handling thousands of concurrent users. BigQuery is optimized for OLAP (online analytical processing) — fast aggregation across billions of rows, scanning specific columns.

The Architecture (Simplified)

Two things make BigQuery fundamentally different from a traditional database:

1. Columnar Storage

Postgres stores data row by row:

Row 1: [42, "Brazil", "BR", "South America", 214000000]
Row 2: [87, "Germany", "DE", "Europe", 83000000]
Row 3: [12, "Japan", "JP", "Asia", 125000000]

BigQuery stores data column by column:

university_count: [42, 87, 12]
country:          ["Brazil", "Germany", "Japan"]
iso_code:         ["BR", "DE", "JP"]
continent:        ["South America", "Europe", "Asia"]
population:       [214000000, 83000000, 125000000]

Why this matters: if your query is SELECT country, SUM(university_count) GROUP BY continent, BigQuery only reads two columns — not all five. On a table with 100 columns and 10 billion rows, this is the difference between seconds and hours.

2. Separated Compute and Storage

In Postgres, the same server handles both storing data and running queries. In BigQuery, they're independent:

  • Storage (Colossus): Your data lives here, replicated across multiple locations automatically
  • Compute (Dremel): Thousands of workers run your query in parallel, then results are assembled

This means you can store petabytes without paying for idle compute. You only pay for compute when queries run.

Cost Implication

On-demand pricing charges $6.25 per TiB scanned (first 1 TiB/month free). BigQuery charges for columns scanned, not rows returned. SELECT * on a wide table is expensive. SELECT country, population is cheap.

The Hierarchy

BigQuery organizes data in a hierarchy you need to internalize:

GCP Project
  └── Dataset (like a schema in Postgres)
        └── Table (like a table in Postgres)
              └── Column (field)

In your ETL pipeline, the ADR-002 decision maps to:

Project: your-gcp-project
  └── Dataset: etl_universities
        ├── Table: universities          (raw enriched data)
        ├── Table: universities_per_country
        ├── Table: universities_per_continent
        └── Table: top10_universities

What BigQuery Is NOT

  • Not a replacement for Postgres. Your app's database stays Postgres. BigQuery is for analytics on data you've already collected.
  • Not a streaming platform. It handles batch loads well; streaming is possible but not its sweet spot.
  • Not a general-purpose compute engine. It runs SQL queries. It doesn't run your Python transform logic (that stays in Polars — see ADR-003).
  • Not free beyond the sandbox. The free tier is generous for small projects, but poorly written queries on large tables can surprise you.
