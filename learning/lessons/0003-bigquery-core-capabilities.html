BigQuery Core Capabilities

You know what BigQuery is (Lesson 001) and how its SQL differs from Postgres (Lesson 002). This lesson covers the platform capabilities you'll discuss in meetings, interviews, and architecture decisions — the things that make you sound like you've been working with BigQuery for months.

1. Pricing — The #1 Interview Topic

BigQuery has two pricing models. Understanding when to use each is the single most valuable piece of platform knowledge.

On-Demand Pricing

You pay per TiB of data scanned by each query. First 1 TiB/month is free.

$6.25 per TiB scanned
Free tier: 1 TiB/month (roughly 65,000 queries on a 1 GB table)

This is what you'll use for most projects. It's simple, predictable at small scale, and requires zero capacity planning.

When on-demand makes sense

Intermittent workloads, ad-hoc analysis, small-to-medium data, dev/test environments, anything under ~1 TiB of monthly query volume. Your ETL pipeline with ~10K rows? On-demand, no question.

Capacity Pricing (Editions)

You reserve compute capacity (slots) and pay per slot-hour, regardless of how much data you scan.

Standard Edition: ~$0.04/slot-hour (auto-scales, no commitment)
Enterprise Edition: ~$0.05/slot-hour (advanced features, SLA)
Enterprise Plus: ~$0.06/slot-hour (highest performance, compliance)
The break-even rule

If you're consistently scanning more than ~1 TiB/month, capacity pricing gets cheaper. Below that, on-demand wins. Most small projects never leave on-demand.

The Safety Valve: max_bytes_billed

Every query can set a scan limit. If the query would exceed it, it fails instead of running up a bill.

-- Set via Python client or console
-- Query fails if it would scan more than 1 GB
max_bytes_billed = 1 * 1024**3  # 1 GB in bytes

This is how you sleep peacefully. Set it on every query in production.

2. Table Types — Choosing the Right One

BigQuery has four table types. Each solves a different problem.

TypeWhat It IsWhen to Use
Native tableData stored in BigQuery's columnar formatDefault choice. Best performance. Your ETL output goes here.
External tableReferences data in GCS, Drive, etc.Query data without loading it. Good for parquet/json files you don't want to duplicate.
ViewSQL query saved as a virtual tableReusable queries, access control, hiding complexity. No stored data.
Materialized viewPre-computed results, auto-refreshedDashboards, repeated expensive aggregations. Trades storage for query speed.

Why ADR-002 Chose Native Tables

Your project loads enriched data + aggregated tables as native tables. The reasoning:

  • Small data (~10K rows) — no benefit from external tables
  • Data is produced by your pipeline, not sitting in GCS
  • Native tables give best query performance
  • Simple setup, no external data source configuration

When External Tables Shine

Imagine you have 50 GB of Parquet files in GCS from another team. You want to run a one-off analysis without copying 50 GB into BigQuery. External table — query it directly, pay only for the scan, leave the data where it is.

3. Partitioning & Clustering — Splitting Data for Speed

These two features solve the same problem differently: reducing how much data each query scans.

Partitioning

Split a table into segments (partitions) based on a column value. Queries that filter on the partition column only scan the relevant partitions.

-- Partition by date (most common)
CREATE TABLE events (
  event_id STRING,
  event_time TIMESTAMP,
  user_id STRING
)
PARTITION BY DATE(event_time);

-- Now this query only scans one day, not the whole table
SELECT COUNT(*) FROM events
WHERE event_time >= '2026-07-01' AND event_time < '2026-07-02';

Partition types:

  • Time-unit — daily (default), hourly, monthly
  • Integer-range — based on an integer column
  • Ingestion-time — partitioned by when data was loaded

Clustering

Sort data within each partition by up to 4 columns. Queries that filter/sort on clustered columns skip irrelevant blocks.

-- Cluster by country within each date partition
CREATE TABLE events (
  event_id STRING,
  event_time TIMESTAMP,
  country STRING,
  user_id STRING
)
PARTITION BY DATE(event_time)
CLUSTER BY country;

Partition vs Cluster — When to Use Which

ScenarioUseWhy
Time-series data, queries filter by datePartitionEliminates entire time ranges from scan
Large table, queries filter by a few categorical columnsClusterSorts data within partitions for faster filtering
BothPartition + ClusterPartition first (coarse), cluster second (fine)
Small table (<1 GB)NeitherFull scan is fast enough; overhead not worth it
Why your project skips both

ADR-002 explicitly decided against partitioning and clustering. Your data is ~10K rows — a full scan takes milliseconds and costs fractions of a penny. Knowing when not to optimize is as valuable as knowing how to optimize.

4. Materialized Views — Pre-Computed Results

A materialized view stores the result of a query on disk. BigQuery keeps it fresh automatically.

-- Create a materialized view for a dashboard
CREATE MATERIALIZED VIEW `etl_universities.universities_per_country`
AS
SELECT country, COUNT(*) as university_count
FROM `etl_universities.universities`
GROUP BY country;

When someone queries the materialized view, BigQuery reads the pre-computed result instead of scanning the source table. Queries are faster, and BigQuery only charges for the small amount of data read from the materialized view.

Materialized View vs Native Table for Aggregations

AspectMaterialized ViewNative Table (your approach)
RefreshAutomatic (BigQuery manages it)Manual (your pipeline rewrites it)
CostStorage + refresh computeLoad cost + query cost
FlexibilityLimited (no JOINs across datasets, no UDFs)Full SQL, full control
ConsistencyEventually consistent (refresh lag)Consistent after pipeline run
Why your project chose native tables

Your aggregations are simple and your data is small. Native tables give you full control, no refresh lag, and no materialized view limitations. The tradeoff is that your pipeline re-computes aggregations on each run — but with ~10K rows, that's negligible.

5. Federated Queries — Query Without Loading

Federated queries let you query data that lives outside BigQuery — GCS files, Drive spreadsheets, even Cloud SQL databases — using standard SQL.

-- Query a Parquet file in GCS without loading it
SELECT * FROM EXTERNAL_QUERY(
  'connection-id',
  'SELECT * FROM `/bucket/data/file.parquet`'
);

-- Or query Google Sheets directly
SELECT * FROM EXTERNAL_QUERY(
  'connection-id',
  'SELECT * FROM `spreadsheet-id`.Sheet1'
);

This is powerful for one-off analysis, data exploration, or when you need to join BigQuery data with external sources. You pay for the data scanned, same as a regular query.

Your ETL pipeline needs: dataEditor (to load tables) + jobUser (to run queries if needed).

Column-Level Security

Hide sensitive columns from certain users without hiding the entire table.

-- Apply it to a column: only users with the Fine-Grained Reader role can see it

Row-Level Security

-- Row-level security policy applies the filter automatically

Use case: multi-tenant dashboards where each client should only see their own data.
