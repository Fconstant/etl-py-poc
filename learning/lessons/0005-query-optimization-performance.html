Query Optimization & Performance

You know what BigQuery is, how its SQL works, its platform capabilities, and how to load data. This lesson covers how to debug and optimize — the skill that separates someone who uses BigQuery from someone who understands it.

1. How BigQuery Executes Queries

When you run a query, BigQuery doesn't just "run it." It goes through a multi-stage pipeline:

Your SQL
  → Parse & validate
    → Query plan (Dremel converts to execution tree)
      → Slot assignment (workers get slices of data)
        → Parallel execution (each worker scans its slice)
          → Result assembly
            → Returned to you

The key insight: BigQuery parallelizes automatically. You don't manage threads, connections, or workers. But understanding this helps you optimize — because you're essentially paying for the work each worker does.

2. The Dry Run — Estimate Cost Before Execution

Before running a query, you can estimate how much data it will scan. This is the single most important cost-control tool.

# Python client
job_config = bigquery.QueryJobConfig(dry_run=True, use_query_cache=False)
query_job = client.query("SELECT * FROM `big_table`", job_config=job_config)

# Bytes scanned (not billed — it's a dry run)
print(f"Will scan: {query_job.total_bytes_processed} bytes")
print(f"Will scan: {int(query_job.total_bytes_processed) / 1024**3:.2f} GiB")
# SQL — set max_bytes_billed to block expensive queries
SELECT *
FROM `big_table`
WHERE date >= '2026-01-01'
-- Set via client: max_bytes_billed = 1 * 1024**3 (1 GB limit)
The workflow

Always dry-run first. If the estimate is too high, rewrite the query. Then set max_bytes_billed as a safety net so no one accidentally runs an expensive query in production.

3. Slot Utilization — Is Your Query starved?

Slots are BigQuery's virtual CPUs. More slots = faster query. On-demand pricing gives you access to a shared pool. Capacity pricing gives you reserved slots.

You can see slot usage in the query execution details:

# After a query runs, check the job statistics
job = client.query("SELECT ...").result()

# Slot milliseconds used
print(f"Slot time: {job.slot_ms} ms")

# Average slots used = slot_ms / elapsed_time_ms
elapsed_ms = (job.ended - job.created).total_seconds() * 1000
avg_slots = job.slot_ms / elapsed_ms
print(f"Average slots: {avg_slots:.1f}")

If your query used 1 slot on average but could use 100, it's because BigQuery is conservative — it allocates slots based on data size and query complexity. Small queries on small tables don't need many slots.

When slots matter

At your scale (~10K rows), slot utilization is irrelevant — queries complete in seconds. This becomes important when you're scanning gigabytes and wondering why a query takes 5 minutes instead of 30 seconds.

4. Execution Details — What BigQuery Tells You

Every query job stores detailed statistics. This is your debugging tool.

INFORMATION_SCHEMA

BigQuery exposes metadata about your projects, datasets, tables, and jobs via SQL views:

-- See your recent queries and their costs
SELECT
  query,
  total_bytes_billed,
  total_bytes_processed,
  creation_time,
  end_time,
  statement_type
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
ORDER BY total_bytes_billed DESC
LIMIT 20;

Query Execution Details

After a query runs, the job object contains a full breakdown:

job = client.query("SELECT ...").result()

# What stages ran
for stage in job.statistics["query"]["executionInfo"]["dag"]["nodes"]:
    print(f"Stage: {stage['id']}")
    print(f"  Records read: {stage.get('recordsRead', 0)}")
    print(f"  Bytes processed: {stage.get('bytesProcessed', 0)}")

# Was the result cached?
print(f"Cache hit: {job.cache_hit}")

5. Query Cache — Free Reads

BigQuery caches query results for 24 hours. If you run the exact same query again, it returns the cached result — free, no bytes scanned.

Conditions for cache hit:

  • Exact same SQL text (including whitespace)
  • Same user project
  • Same destination table (if applicable)
  • No changes to referenced tables since last run
# Check if a query hit cache
job = client.query("SELECT ...").result()
if job.cache_hit:
    print("Free read — cache hit!")
else:
    print(f"Scanned: {job.total_bytes_processed} bytes")
Practical impact

If you run the same dashboard query every hour, you pay for it once — the other 23 runs are free (as long as the underlying data hasn't changed). This is a massive cost advantage for read-heavy workloads.

6. Optimization Patterns Beyond the Basics

Lesson 002 covered the basics: don't SELECT *, filter early, LIMIT doesn't save cost. Here are the intermediate patterns.

Approximate Aggregations

When exact counts aren't necessary, BigQuery can approximate much faster:

-- Exact distinct count (scans everything)
SELECT COUNT(DISTINCT user_id) FROM events;

-- Approximate (1-2% error, but 10x faster)
SELECT COUNT(DISTINCT user_id) FROM events;
-- vs
SELECT approx_count_distinct(user_id) FROM events;

ANALYZE for Statistics

BigQuery collects column statistics automatically, but you can force an update:

ANALYZE TABLE `your-project.etl_universities.universities`;

This helps BigQuery's query planner make better decisions about join order and data skipping.

Clustered Table Queries

If you have a clustered table (Lesson 003), queries that filter on clustered columns automatically skip irrelevant blocks:

-- Table clustered by country, partitioned by date
-- This query only scans European data blocks
SELECT COUNT(*)
FROM events
WHERE country = 'Germany'
  AND event_date = '2026-07-19';

Temp Tables for Complex Logic

Use temp tables instead of CTEs when you need the same data multiple times:

-- CTE = re-scanned each time (billed twice)
WITH data AS (SELECT * FROM big_table)
SELECT COUNT(*) FROM data
UNION ALL
SELECT SUM(amount) FROM data;

-- Temp table = scanned once (billed once)
CREATE TEMP TABLE tmp AS SELECT * FROM big_table;
SELECT COUNT(*) FROM tmp UNION ALL SELECT SUM(amount) FROM tmp;

Struct Flattening

When loading nested JSON/Arrow data, BigQuery stores structs natively. Query them with dot notation instead of UNNEST for simpler syntax:

-- If address is a STRUCT column:
SELECT address.city, address.country FROM users;

-- vs UNNEST (only needed for arrays, not structs)
SELECT addr.city FROM users, UNNEST(addresses) AS addr;

7. Performance Debugging Workflow

When a query is slow or expensive, follow this sequence:

  1. Dry run — check estimated bytes scanned
  2. Check execution details — which stage took the most time?
  3. Check slot utilization — was the query slot-starved?
  4. Check partition/clustering — is the table structured for this query pattern?
  5. Check data volume — has the table grown since last measured?
  6. Check cache — is this query supposed to be cached? Why isn't it?
# Quick diagnostic query
SELECT
  query,
  total_bytes_processed / 1024 / 1024 AS mb_scanned,
  total_bytes_billed / 1024 / 1024 AS mb_billed,
  slot_ms / 1000 AS slot_seconds,
  TIMESTAMP_DIFF(end_time, creation_time, SECOND) AS wall_seconds,
  cache_hit
FROM `region-us`.INFORMATION_SCHEMA.JOBS_BY_PROJECT
WHERE creation_time > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
ORDER BY total_bytes_billed DESC
LIMIT 10;
