Python Client & Loading Data

You understand the platform. Now the mechanical part: getting data from Polars into BigQuery. This lesson is concrete — code you'll copy into your project.

The Flow

Polars DataFrame
  → PyArrow Table  (in-memory, columnar)
    → BigQuery API  (upload + load job)
      → BigQuery Table  (stored in Colossus)

PyArrow is the bridge. Polars speaks Arrow natively; BigQuery's Python client accepts Arrow tables. No CSVs, no JSON, no serialization overhead.

1. Authentication

BigQuery requires a GCP identity. For a service like your ETL pipeline (no human logging in), you use a service account.

What You Need

  1. A GCP project with BigQuery API enabled
  2. A service account with roles/bigquery.dataEditor + roles/bigquery.jobUser
  3. A JSON key file for that service account

Application Default Credentials (ADC)

The Google Cloud client library uses ADC to find credentials automatically. The simplest way:

# Set this environment variable before running your pipeline
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/service-account-key.json"

The BigQuery client picks it up automatically — no code needed for auth.

Never commit the key file

Add the JSON key to .gitignore immediately. If it leaks, anyone can use your BigQuery resources on your bill.

Creating the Client

from google.cloud import bigquery

client = bigquery.Client(
    project="your-gcp-project-id",
    # credentials auto-discovered from ADC
)

One client, reusable across all operations. Create it once at the top of your load function.

2. Polars → Arrow → BigQuery

This is the core pattern. Three lines of code:

import polars as pl
from google.cloud import bigquery

# Your Polars DataFrame
df: pl.DataFrame = ...  # from transform stage

# Step 1: Convert to Arrow (zero-copy — Polars is built on Arrow)
arrow_table = df.to_arrow()

# Step 2: Load into BigQuery
job = client.load_table_from_object(
    arrow_table,
    destination="your-project.etl_universities.universities",
    job_config=bigquery.LoadJobConfig(
        write_disposition="WRITE_TRUNCATE",
    ),
)

# Step 3: Wait for the job to finish
job.result()  # blocks until load completes

to_arrow() is zero-copy because Polars is built on Apache Arrow internally. No data is serialized — the Arrow format is shared between Polars and the BigQuery client.

3. Write Dispositions

What happens when the target table already exists?

DispositionBehaviorWhen to Use
WRITE_TRUNCATEOverwrite the table entirelyBatch pipelines that re-run from scratch (your case — ADR-002)
WRITE_APPENDAdd rows to the existing tableIncremental loads, streaming
WRITE_EMPTYFail if table already has dataSafety check for one-time loads

Your pipeline uses WRITE_TRUNCATE because each run produces a complete snapshot of the data. Old results are replaced, not appended.

4. Schema Mapping

BigQuery needs to know your column types. When loading from Arrow, it infers them automatically — but you can override.

job_config = bigquery.LoadJobConfig(
    write_disposition="WRITE_TRUNCATE",
    autodetect=False,  # don't guess — be explicit
    schema=[
        bigquery.SchemaField("university_name", "STRING"),
        bigquery.SchemaField("country", "STRING"),
        bigquery.SchemaField("iso_code", "STRING"),
        bigquery.SchemaField("continent", "STRING"),
        bigquery.SchemaField("population", "INT64"),
        bigquery.SchemaField("currency_name", "STRING"),
    ],
)

Polars → BigQuery Type Mapping

Polars TypeArrow TypeBigQuery Type
Utf8 / StringstringSTRING
Int8 / Int16 / Int32 / Int64int8–int64INT64

Autodetect vs explicit schema

For your project, autodetect works fine — your schema is simple. In production, explicit schema prevents surprises: a column that was INT silently becoming STRING because of a null value in the source.

5. Your to_bigquery() Function

Tying it all together — the function that goes into your load layer:

from google.cloud import bigquery
import polars as pl


def to_bigquery(
    dfs: dict[str, pl.DataFrame],
    project_id: str,
    dataset: str,
) -> None:
    """Load multiple DataFrames into BigQuery tables.

    Args:
        dfs: Mapping of table_name → DataFrame
        project_id: GCP project ID
        dataset: BigQuery dataset name
    """
    client = bigquery.Client(project=project_id)

    for table_name, df in dfs.items():
        table_id = f"{project_id}.{dataset}.{table_name}"
        arrow_table = df.to_arrow()

        job = client.load_table_from_object(
            arrow_table,
            destination=table_id,
            job_config=bigquery.LoadJobConfig(
                write_disposition="WRITE_TRUNCATE",
            ),
        )
        job.result()  # wait for completion

        print(
            f"Loaded {df.shape[0]} rows into {table_id}"
        )

Usage from your pipeline:

to_bigquery(
    dfs={
        "universities": enriched_df,
        "universities_per_country": per_country_df,

    },
    project_id="your-gcp-project",
    dataset="etl_universities",

7. Error Handling

Common failures and how to handle them:

from google.api_core.exceptions import GoogleAPIError

try:
    job.result()
except GoogleAPIError as e:
    print(f"Load failed: {e}")
    # Job details for debugging
    print(f"Job ID: {job.job_id}")
    print(f"Errors: {job.errors}")
Job.result() blocks forever

If the load job hangs (network issue, quota exceeded), job.result() blocks indefinitely. Set a timeout: job.result(timeout=300) (5 minutes). For small datasets, loads complete in seconds.
