GoogleSQL Basics

You know Postgres SQL. BigQuery uses GoogleSQL — ANSI-compliant, but with differences that will bite you if you copy-paste from Postgres. This lesson covers the syntax that changes, the cost patterns baked into query writing, and the idiom shifts you need.

The Big Four Differences

1. Backticks, Not Quotes

In Postgres, you escape reserved words and special identifiers with double quotes. In BigQuery, you use backticks:

-- Postgres
SELECT "user", "order" FROM "my-schema"."my-table";

-- BigQuery (GoogleSQL)
SELECT `user`, `order` FROM `my-dataset.my-table`;
Rule of thumb

Double quotes in Postgres → backticks in BigQuery. Always. Column names with spaces, hyphens, or reserved words all need backticks.

2. Table References Use Dots, Not Schemas

Postgres: schema.table. BigQuery: dataset.table. The project can be prepended with a dot too:

-- Postgres
SELECT * FROM analytics.users;

-- BigQuery
SELECT * FROM `etl_universities.universities`;

-- With explicit project (rare, usually set as default)
SELECT * FROM `my-project.etl_universities.universities`;

3. No PostgreSQL-Specific Functions

Functions you use daily in Postgres don't exist in BigQuery. Here's the cheat sheet:

4. Data Types Are Simpler

Postgres has many integer types. BigQuery collapses them:

-- Postgres: SMALLINT, INTEGER, BIGINT, SERIAL, etc.
-- BigQuery: just INT64

-- Postgres: FLOAT, DOUBLE PRECISION, NUMERIC
-- BigQuery: FLOAT64 (float) and NUMERIC (exact decimal)
Gotcha: Division

In Postgres, 7 / 2 returns 3 (integer division). In BigQuery, 7 / 2 returns 3.5 (FLOAT64). Use DIV(7, 2) for integer division. This will silently change your aggregation results.

Cost-Aware Query Patterns

These patterns directly impact your BigQuery bill. On-demand pricing: $6.25 per TiB scanned.

Pattern 1: Never SELECT *

-- BAD: Scans every column ($$$)
SELECT * FROM `etl_universities.universities`;

-- GOOD: Scan only what you need ($)
SELECT university_name, country, continent
FROM `etl_universities.universities`;

On a 100-column table, selecting 3 columns costs 97% less than SELECT *.

Pattern 2: Filter Early

-- BAD: Join then filter
SELECT u.university_name, c.population
FROM `etl_universities.universities` u
JOIN `etl_universities.countries` c ON u.iso_code = c.iso_code
WHERE c.continent = 'Europe';

-- GOOD: Filter before join (subquery reduces join size)
SELECT u.university_name, c.population
FROM (
  SELECT university_name, iso_code
  FROM `etl_universities.universities`
) u
JOIN (
  SELECT iso_code, population
  FROM `etl_universities.countries`
  WHERE continent = 'Europe'
) c ON u.iso_code = c.iso_code;

Pattern 3: LIMIT Doesn't Reduce Cost

-- This STILL scans the entire table:
SELECT * FROM `etl_universities.universities`
ORDER BY university_name
LIMIT 10;

-- You're billed for every column in every row,
-- even though only 10 are returned.
Trap for Postgres devs

In Postgres, LIMIT is a performance hint — the planner can stop early. In BigQuery, LIMIT is applied after the full scan. You still pay for the whole table. The one exception: on clustered tables, LIMIT can stop scanning early once enough blocks are found.

Pattern 4: Avoid Repeated CTEs

-- BAD: CTE used twice = scanned twice (billed twice)
WITH data AS (
  SELECT * FROM `big_table`
)
SELECT COUNT(*) FROM data
UNION ALL
SELECT SUM(amount) FROM data;

-- GOOD: Use a temp table if you need the data twice
CREATE TEMP TABLE tmp AS
SELECT * FROM `big_table`;

SELECT COUNT(*) FROM tmp
UNION ALL
SELECT SUM(amount) FROM tmp;

CTEs in BigQuery are inlined (like macros), not materialized. Each reference re-scans the source.

Pattern 5: Use SAFE Functions to Avoid Errors

-- This fails on division by zero:
SELECT amount / 0 FROM `transactions`;

-- This returns NULL on division by zero (safe):
SELECT SAFE_DIVIDE(amount, 0) FROM `transactions`;

-- SAFE_CAST returns NULL instead of failing:
SELECT SAFE_CAST(invalid_col AS INT64) FROM `data`;

The GoogleSQL Extras

GoogleSQL has features Postgres doesn't:

QUALIFY — Filter Window Functions Inline

-- Postgres: subquery needed
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY country ORDER BY population DESC) as rn
  FROM universities
) WHERE rn = 1;

-- BigQuery: QUALIFY makes it clean
SELECT *, ROW_NUMBER() OVER (PARTITION BY country ORDER BY population DESC) as rn
FROM universities
QUALIFY rn = 1;

SELECT * EXCEPT — Drop Columns From Results

-- Select everything EXCEPT the auto-generated row_number
SELECT * EXCEPT(rn) FROM (
  SELECT *, ROW_NUMBER() OVER (...) as rn
  FROM universities
) WHERE rn = 1;

UNNEST — Explode Arrays Into Rows

-- If you have an ARRAY column:
SELECT tag FROM `posts`, UNNEST(tags) AS tag;

-- Equivalent in Postgres would be:
-- SELECT unnest(tags) FROM posts; (similar but syntax differs)
